# ADR-002

## UI — Webview в стиле Claude Code

Окно диалога рисуем на Webview (как и планировалось в PROJECT.md), но
оформляем в стиле расширения Claude Code, а не имитируем Telegram.

Рассмотренные альтернативы

- Chat Participant API — нативно, но turn-based модель ассистента плохо
  ложится на живой мессенджер с многими чатами; возможна зависимость от Copilot.
- TreeView + редактор-транскрипт — нативно, но нет нужного вида.

Выбор: Webview.

Ориентир — расширение **Claude Code for VS Code** (снят дизайн вживую).

Принципы оформления

- только цвета из темы VS Code (`var(--vscode-*)`) + сдержанный акцент;
- чистый вертикальный список сообщений, без пузырей Telegram;
  входящее — маркер-точка ● + автор + текст; своё — в лёгкой скруглённой рамке;
- композер как у Claude Code: скруглённый блок, поле ввода + панель
  (`+`/`/` слева, стрелка отправки справа);
- markdown в сообщениях, слэш-команды и @-упоминания — позже.

Расположение (уточнено, повторяет Claude Code for VS Code):

- список чатов — в сайдбаре Yapper (нативный TreeView);
- открытый диалог — **вкладка редактора во всю ширину** (`WebviewPanel`),
  одна панель переиспользуется между чатами, имя вкладки = название чата,
  контент центрирован колонкой.

Ранее диалог планировался боковой панелью — заменено на вкладку редактора,
чтобы точно повторить Claude Code for VS Code.

# ADR-003

## Хранение сессии и учётных данных Telegram

`api_id`, `api_hash` и строка сессии GramJS (`StringSession`) хранятся в
VS Code **SecretStorage**, а не в settings.json.

Причины

- секреты не попадают в открытый settings.json и в синхронизацию настроек;
- сессия переживает перезапуск VS Code (требование MVP);
- один механизм для всех секретов.

`api_id`/`api_hash` пользователь получает на my.telegram.org и вводит один раз
при первом входе; дальше переиспользуются из SecretStorage.

Логин: **только по QR-коду** (`signInUserWithQrCode`). Показываем QR в
webview-панели (`tg://login?token=...`), пользователь сканирует его в мобильном
Telegram (Настройки → Устройства → Подключить устройство). Пароль 2FA
запрашивается через `showInputBox`, только если он включён. Телефон+код не
используем — QR удобнее. `api_id`/`api_hash` всё равно нужны (требование MTProto),
вводятся один раз.

Переподключение на старте — `client.connect()` + `checkAuthorization()`.

# ADR-004

## Провайдер-независимое форматирование сообщений

Форматирование текста (жирный/курсив/код/ссылка/упоминание/спойлер/цитата/…)
описываем **общей моделью сущностей** `MessageEntity` в `providers/types.ts`, а
не хардкодим в UI под конкретный мессенджер.

Причины

- у Telegram/WhatsApp/Slack/Discord форматирование выражается одинаково —
  диапазонами (offset+length) над плоским текстом;
- каждый провайдер маппит свой формат в общие `MessageEntity`, а webview
  рендерит единообразно (`renderRichText`);
- при добавлении новых мессенджеров UI менять не нужно.

Модель: `{ type, offset, length, url?, language? }`, оффсеты в UTF-16
(индексы JS-строки). Telegram-провайдер конвертирует `Api.MessageEntity*` →
`MessageEntity`. Если entities нет — фолбэк на регулярку (URL/@упоминания).

# ADR-005

## Медиа: фото/видео в лайтбоксе; звук видео — внешний плеер; голосовые отложены

Webview VS Code (Chromium в Electron на этой сборке) **не декодирует
аудиодорожку**: видео проигрывается без звука, голосовые (Opus/OGG) не
воспроизводятся вообще.

Решения

- **Изображения** — открываются во встроенном **лайтбоксе** (оверлей поверх
  диалога), полноразмерный `<img>` через webview-URI. Клик по подложке / Esc /
  крестик — закрыть.
- **Видео** — тоже в лайтбоксе (`<video controls autoplay>`), но т.к. звука в
  webview нет, там есть кнопка **«Открыть со звуком»** → внешний системный плеер
  (`env.openExternal`), где кодеки полные. Файл кешируется на диск, отдаётся
  webview через `asWebviewUri` (CSP `media-src`/`img-src` = cspSource,
  `storageUri` в `localResourceRoots`).
- **Голосовые** — инлайн-плеер убран (пробовали `data:`-URL и webview file-URI,
  оба молчат). Пока голосовое доступно как **скачиваемый файл** (плашка), имя
  `Голосовое сообщение.ogg`. Внешний фолбэк на macOS упирается в отсутствие
  приложения для `.ogg`, поэтому автоматически его не открываем.

Отложено (см. [NOT_NOW.md](NOT_NOW.md)): инлайн-прослушивание голосовых.
Возможные пути на будущее — транскод в поддерживаемый контейнер или
воспроизведение вне webview.

Провайдер отдаёт настоящий `mimeType`/расширение из документа Telegram
(`MediaFile.mime`), чтобы контейнеры не мислейблились.

# ADR-006

## Уведомления: badge + toast; звук отложен

Реализовано

- **Badge** — суммарный счётчик непрочитанных на иконке Yapper через
  `TreeView.badge` (для этого перешли с `registerTreeDataProvider` на
  `createTreeView`). `ChatTreeProvider` эмитит `onDidChangeBadge` при пересборке
  списка.
- **Toast** — `window.showInformationMessage` на входящее сообщение с кнопкой
  «Открыть». Подавляется, если пользователь уже смотрит этот чат
  (`ConversationPanel.isViewing`). Управляется настройками
  `yapper.notifications.enabled` и `yapper.notifications.showPreview`.

Отложено: **Sound**

У VS Code нет публичного API проигрывания произвольного звука из extension host
(внутренние accessibility signals недоступны расширениям). Единственный путь —
постоянно живой скрытый webview только ради звука; это расточительно и хрупко
(см. проблемы с аудио-кодеками в ADR-005). Вернёмся, если появится API или
реальная необходимость.

## Share Code / Diff: цель — открытый чат, код как Pre-entity

Команды `yapper.shareSelection` (выделение или весь файл) и
`yapper.shareDiff` (`git diff`) отправляют код в чат, **открытый в панели**

- Лимит Telegram 4096 символов: длинный код/diff обрезается с пометкой
  «(обрезано)». Отправка большого diff файлом — позже.

Точки входа: контекст-меню редактора (Share Selection), инлайн-кнопка в SCM

# ADR-008

## Локализация: английский базовый, три слоя

Базовый (исходный) язык — **английский**; русский — перевод. Так работает
стандартный механизм VS Code (язык следует настройке редактора,
`vscode.env.language`), и международным пользователям по умолчанию — EN.

Три слоя строк:

- **package.json** (декларативное) — строки вынесены в `%ключи%`,
  значения в `package.nls.json` (EN, дефолт) и `package.nls.ru.json` (RU).
- **extension host** (TS) — `vscode.l10n.t("English source", ...args)`; переводы
  в `l10n/bundle.l10n.ru.json` (ключ = английский исходник). Объявлено
  `"l10n": "./l10n"` в package.json. Даты — `toLocaleString(vscode.env.language)`.
- **webview** (conversation.js) — не имеет доступа к `vscode.l10n`, поэтому host
  собирает словарь и инъектит его как `window.L10N` в HTML; скрипт читает `L.*`.
  Плейсхолдеры (`{0}`) прокидываются через `t("… ({0})", "{0}")` и `.replace`.

Добавление языка = один файл `package.nls.<lang>.json` +
`l10n/bundle.l10n.<lang>.json`. Ресурсы (`l10n/`, `package.nls*.json`, `media/`)
не исключены в `.vscodeignore` — попадают в VSIX.

Медиа-подписи («🖼 Фото», «📎 Файл», имя «Голосовое сообщение.ogg») тоже
локализованы — это часть контента, видимого пользователю.

# ADR-009

## Мульти-провайдер: переключение активного (MVP)

Второй мессенджер (WhatsApp) добавлен так, что провайдеры **переключаются**, а
не показываются в едином дереве одновременно. В UI активен один провайдер;
команда `yapper.switchMessenger` меняет его.

Причины

- минимальный риск: не нужно протаскивать `providerId` в `Chat`/`Message` и
  маршрутизацию через всё дерево/панель/вебвью;
- достаточно, чтобы доказать провайдер-агностичность ядра.

Реализация

- realtime + жизненный цикл вынесены в интерфейс `Messenger extends
  MessengerProvider` (`connected`/`login`/`logout`/`init`/`dispose`/`onMessage`/
  `onConnectionChange`/…). Данные-методы остались в `MessengerProvider`, чтобы
  `MockProvider` не тянул `vscode` (тесты).
- `extension.ts` держит реестр провайдеров и активный (в `globalState`);
  подписан на события **всех**, но реагирует только на активный (`p === active`).
- `ChatTreeProvider`/`ConversationPanel` получили `setProvider()`; при
  переключении открытый диалог закрывается, кэши чистятся.

Отложено (см. [NOT_NOW.md](NOT_NOW.md)): **единое дерево** обоих провайдеров
одновременно + `providerId` в моделях.

# ADR-010

## WhatsApp через Baileys, text-only, событийный стор

Провайдер WhatsApp построен на **`@whiskeysockets/baileys`** (не официальном
API): чистый WebSocket без Chromium, QR-логин, доступ к личным чатам.

Причины

- официального API для личных переписок у WhatsApp нет (Cloud API — только для
  бизнес-рассылок); Baileys — единственный реалистичный путь, архитектурно
  близкий к Telegram (QR, realtime);
- **бан-риск** (против ToS WhatsApp) принят осознанно.

Особенности

- данные **событийные**, не request/response: чаты/контакты/история приходят в
  `messaging-history.set`, `chats.*`, `messages.upsert`. Провайдер держит
  собственный in-memory стор и отвечает из него; обновление списка чатов — через
  переиспользование `onConnectionChange` как «refresh»-сигнала (дебаунс).
- сессия — `useMultiFileAuthState` в папке под `globalStorageUri`.
- **realtime** родной (`messages.upsert`, type `notify`); эхо своих отправок
  фильтруется по id.
- **Baileys 7 — ESM-only**, проект CJS → `moduleResolution: Bundler` для `tsc`
  (сборку делает esbuild). Опциональные медиа-пиры (`sharp`/`jimp`/…) помечены
  `external`.

MVP — **text-only**: без медиа/поиска/профилей (см. NOT_NOW). Пагинация истории
(ADR-011) и read-receipts (ADR-012) добавлены позже.

# ADR-011

## WhatsApp: пагинация истории через on-demand-догрузку

Провайдер отдаёт историю из собственного стора, но при первом коннекте WhatsApp
присылает лишь недавние сообщения. Чтобы скролл вверх подгружал более старые,
реализуем провайдер-агностичный `getMessagesBefore`.

Решение

- `getMessagesBefore(chatId, beforeId)` сначала отдаёт старые сообщения, уже
  лежащие в сторе; если их нет — запрашивает у сервера `fetchMessageHistory`
  (Baileys) от самого старого известного сообщения.
- Ответ **асинхронный и событийный**: старая история прилетает в
  `messaging-history.set` (syncType `ON_DEMAND`), иногда — в `messages.upsert`.
  Провайдер ждёт прихода чанка для нужного чата (одноразовый waiter с таймаутом
  8 c) и затем отдаёт страницу; пустой результат = конец истории для UI.

Особенности

- **Развязка с `MESSAGE_CAP`**: обрезание истории убрано из in-memory стора
  (иначе догруженное тут же выкидывалось) — лимит применяется только при записи
  на диск. За сессию чат может держать в памяти больше сообщений.
- у сервера запрашивается больше (`HISTORY_FETCH = 50`), чем отдаётся за шаг
  (`historyPageSize = 25`): излишек отдаётся следующим шагом локально, а
  эвристика «короткая страница = конец» срабатывает честно.
- ключ для запроса реконструируется из хранимого сообщения
  (`remoteJid`/`id`/`fromMe`/`participant`); для LID-чатов используется
  канонический (PN) jid — редкий edge, принят осознанно.

# ADR-012

## WhatsApp: read-receipts через per-message `onMessageEdited`

Галочки ✓/✓✓ (`Message.status` = `sent`/`read`) для исходящих сообщений — как
уже сделано у Telegram. Модель статусов провайдеро-независима, UI-часть готова.

Решение

- модель Yapper двусостоянийная (✓ / ✓✓, без цвета), а WhatsApp показывает ✓✓
  уже на **доставке**; поэтому маппим `SERVER_ACK(2) → sent (✓)`,
  `DELIVERY_ACK/READ/PLAYED(3/4/5) → read (✓✓)`. Только для **исходящих в личке**
  (в группах — undefined, как у Telegram);
- при отправке ставим `sent` **оптимистично** (успешный `sock.sendMessage` =
  дошло до сервера), чтобы ✓ появлялась сразу, не дожидаясь события;
- живые изменения — два события через общий `applyStatus`:
  `messages.update` (агрегатный статус) и `message-receipt.update` (per-peer
  доставка/прочтение). Обновляем хранимое сообщение и шлём `onMessageEdited`,
  по которому webview перерисовывает конкретное сообщение.

Старая история

- у сообщений, сохранённых до фичи (или без синхронизированного статуса), поля
  `status` нет. При отдаче (`getMessages`/`getMessagesBefore`) исходящие в личке
  без статуса **по умолчанию помечаются ✓✓** (в истории → были доставлены) —
  это display-only дефолт, стор не меняется, живой апдейт всегда важнее.

Ограничение

- статус **READ (прочитано)** на linked-device через Baileys стабильно не
  доходит — до нас приходит максимум `DELIVERY_ACK`. Поэтому «доставлено» и
  «прочитано» сведены к одному ✓✓ (сознательно; отдельного «синего» состояния
  нет). Проверено по логам событий.

Почему не `onReadOutbox` (как у Telegram)

- у Telegram статус приходит моделью «прочитано до maxId», и UI-путь
  `applyReadOutbox` сравнивает id **численно** (`Number(m.id) > max`);
- у WhatsApp id сообщений **строковые/непорядковые**, а статус приходит
  по-сообщению — поэтому естественный путь `onMessageEdited` (перерисовка
  конкретного сообщения), а не maxId-модель.

Особенности

- только forward-переходы (`undefined → sent → read`), без даунгрейда;
- обновлённый статус персистится со стором — галочка переживает переоткрытие
  чата и рестарт.

# ADR-013

## WhatsApp: mute — read-only (уважаем мьюты)

Как и у Telegram, замьюченные чаты не должны пикать тостами
(`extension.ts` → `maybeNotify` спрашивает `isChatMuted`). Но mute-тумблер в
UI живёт только в карточке профиля, а её у WhatsApp (text-only) нет.

Решение

- делаем **read-only**: провайдер читает `Chat.muteEndTime` из Baileys
  (`chats.upsert`/`update`/history), хранит его в сторе (`StoredChat.mutedUntil`,
  переживает рестарт) и учитывает в `isChatMuted` и `Chat.muted`;
- `muteActive(muteEndTime, now)` — чистый хелпер: 0/нет → не мьют, `<0` →
  «навсегда», иначе сравнение с now (таймстемп нормализуется сек/мс);
- **записи нет** (`chatModify` умеет, но вызывать неоткуда без UI-точки).
  Замьютить/размьютить из Yapper — позже, вместе с карточкой профиля (там же,
  где у Telegram). Пока: замьютил в приложении WhatsApp → тихо и в Yapper.

Почему так

- read-only закрывает главную пользу mute (тишина уведомлений) без нового UI и
  без риска; тумблер — отдельная, уже отложенная поверхность (профили).

# ADR-014

## WhatsApp: editor-парити — `sendCode` / `sendFile`

Главная фишка проекта (ADR-007) — кинуть код/дифф/файл из редактора в чат —
на WhatsApp не работала: `ConversationPanel.shareCode/shareFile` требуют
`provider.sendCode`/`sendFile`, а WhatsApp-провайдер их не имел (Share
Selection/Diff/File бросали «not supported»).

Решение

- **sendCode**: WhatsApp-моноширинный формат — тройные бэктики, поэтому
  получателю шлём `` ```<код>``` ``. Но в наш webview кладём **сырой код** с
  entity `pre` (offset 0..len, +language), чтобы рисовался код-блок, а не
  литеральные бэктики. Эхо гасим через `sentIds`.
- **sendFile**: читаем файл с диска и шлём `document` (Baileys требует
  `mimetype` — берём из расширения через `mimeOf`, исходники → `text/plain`,
  иначе `application/octet-stream`). Возвращаем сообщение с `file`-плашкой.
- общий `recordOutgoing` собирает наше исходящее (id/ts/`outgoing`/оптимистичный
  `sent`-статус в личке), вставляет в стор через `insertMessage` и гасит эхо.

Особенности

- документы не тянут media-пиров (`sharp`/`jimp` externalized) — грузится сырой
  буфер, без генерации превью;
- `topicId` игнорируется (у WhatsApp нет форум-топиков).

# ADR-015

## Синхронизация: `resync()` на кнопке Обновить

У WhatsApp `getChats` читает **локальный стор**, поэтому кнопка «Обновить»
лишь перерисовывала кэш (в отличие от Telegram, где `getChats` ходит в сеть).
А WhatsApp историю повторно не присылает — пропущенное при офлайне приходит
только при **переподключении сокета**.

Решение

- опциональный `resync()` в интерфейсе `Messenger`; кнопка «Обновить» зовёт
  `active.resync?.()` перед перерисовкой дерева;
- WhatsApp: `resync` = переподключить сокет (WhatsApp повторно доставит
  pending-уведомления). Telegram не реализует — его `getChats` и так синкает;
- **защита от частых нажатий**: кулдаун 15 c + guard от параллельного запуска +
  no-op без сессии (переподключаться слишком часто — риск троттлинга WhatsApp).

Ограничение

- resync возвращает только то, что WhatsApp считает **непоставленным** этому
  устройству. Сообщения, которые сокет уже получил (и, например, отбросил
  багом), сервер повторно не шлёт — их так не вернуть. Ср. фикс разворачивания
  обёрток, который не даёт их ронять впредь.

# ADR-016

## WhatsApp: входящие медиа

Заменяем заглушки «[media]» реальными картинками/файлами (как у Telegram,
провайдеро-агностичный UI уже это умеет: `getMedia` — превью, `getMediaFile` —
полная загрузка, `hasImage`/`mediaKind`/`file` в модели).

Решение

- **маппинг** в `toMessage` через `mediaInfo` (по развёрнутому контенту):
  image → `photo`, video → `video`/`gif` (по `gifPlayback`), sticker →
  `sticker`, document/audio → `file`-плашка; caption становится текстом;
- **превью** (`getMedia`) — берём встроенный `jpegThumbnail` (без загрузки).
  У стикеров его нет → качаем сам (небольшой) стикер;
- **полная загрузка** (`getMediaFile`) — `downloadMediaMessage(raw, "buffer")`
  с `reuploadRequest: sock.updateMediaMessage` (перезапрос протухших URL);
  mime/расширение/имя — из полей сообщения (`extFromMime`/`extOf`);
- **список чатов** — `previewOf`: текст, иначе локализованная метка
  (🖼 Photo / 🎥 Video / 📎 имя).

Особенности / ограничение

- нужен «сырой» `WAMessage` (media-ключи) — держим его в **in-memory** кэше
  `rawMessages` (как messageCache у Telegram); на диск ключи не пишем;
- **после рестарта** превью/скачивание для уже сохранённых медиа не работают,
  пока сообщение не придёт заново или не подгрузится пагинацией (тогда `raw`
  снова в кэше). Telegram этим не страдает, т.к. его `getMessages` каждый раз
  ходит в сеть; у WhatsApp `getMessages` читает локальный стор. Приемлемо для
  MVP; при необходимости — персист миниатюр/дескрипторов позже;
- документы качаются без media-пиров (`sharp`/`jimp` externalized).

# ADR-017

## WhatsApp: аватары чатов

Фото чата в шапке диалога (`getAvatar`, как у Telegram; UI уже готов —
`ConversationPanel` подставляет его в `load`).

Решение

- источник фото: сначала `Contact.imgUrl` из синка (надёжно, без запроса),
  иначе `sock.profilePictureUrl(jid, "preview", timeout)` → `fetch` → data URL;
- кэш `avatars` по jid, **включая промахи**, чтобы флейковое фото не
  перезапрашивалось каждое открытие; свежий `imgUrl` сбрасывает кэш
  (`storeContact`); чистится в `clearStore`;
- аватар грузится **off the critical path**: `ConversationPanel` постит `load`
  сразу, а фото стримит в шапку отдельным `headerAvatar` — медленный/зависший
  запрос не блокирует «loading messages».

Ограничение (проверено на живом аккаунте)

- `profilePictureUrl` для **1:1 ненадёжен**: стабильные `Timed Out` /
  `item-not-found`, тогда как **группы** работают (у них фото из метаданных
  группы). Если WhatsApp не кладёт `imgUrl` в синк — аватара 1:1 не будет.
  Это ограничение WhatsApp/Baileys; 1:1-аватары — **best-effort**.

Границы

- только **аватар чата** (шапка). Дерево чатов — на `ThemeIcon` (как у
  Telegram). Пер-сообщенческие аватары авторов в группах (`Message.avatar`) —
  позже (нужен async-проход по отправителям при загрузке).

# ADR-018

## Отправка изображения из буфера обмена — через paste в вебвью

Кинуть скриншот из буфера прямо в чат. У **extension host** VS Code буфер
обмена **только текстовый** (`vscode.env.clipboard` = `readText`/`writeText`),
картинку оттуда не достать без нативного бинарника на каждую ОС
(`pngpaste`/PowerShell/`xclip`) — хрупко. Поэтому идём через **вебвью**
(Chromium): он читает картинку из браузерного события `paste`.

Реализация

- **вебвью** (`conversation.js`): слушаем `paste` на `#input` (и `drop` на
  композере — бесплатный drag-and-drop). Первый item с `image/*` →
  `getAsFile()` → `FileReader.readAsDataURL` → превью-чип `#image-bar` над
  вводом (по образцу `#reply-bar`). Текст в поле становится **подписью**;
  Send шлёт `{ type:"sendImage", dataUrl, mime, caption }`. `updateSendState`
  разрешает отправку картинки без текста; `Escape`/смена чата — сброс.
- **хост** (`ConversationPanel.handleSendImage`): декодирует data URL →
  пишет во временный файл в `storageUri` (`paste_<ts>.<ext>`) → зовёт
  `provider.sendImage`. **Провайдеро-агностичное превью своего сообщения**:
  локальный файл засеивается в `mediaCache` (инлайн, через поле `media` в
  `append`) и в `fileCache` (лайтбокс) — без похода в `getMedia`.
- **интерфейс**: новый опциональный `sendImage(chatId, filePath, caption?,
  topicId?)`. Отдаём **фото**, а не документ (иначе теряется инлайн-превью;
  `sendFile` осознанно шлёт документом для исходников).
- **Telegram**: тот же `client.sendFile` **без** `forceDocument` → инлайн-фото
  (`toMessage` проставляет `hasImage`/`mediaKind:"photo"`).
- **WhatsApp**: `sock.sendMessage(jid, { image, caption })` + `recordOutgoing`;
  raw кэшируется в `rawMessages` для повторного открытия.

Особенности / границы

- используем именно событие `paste`, не `navigator.clipboard.read()` —
  последнее в вебвью обычно заблокировано.
- CSP не трогаем: `img-src` уже разрешает `data:`, локальный файл идёт через
  `asWebviewUri` (cspSource), сети нет.
- у WhatsApp сгенерированной миниатюры может не быть (media-пиры `sharp`/`jimp`
  externalized, см. ADR-016), но своё превью берётся из локального файла, так
  что показывается всегда; у собеседника фото уходит корректно.
